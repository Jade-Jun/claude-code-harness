# 프로젝트 가이드

Next.js App Router + 클린 아키텍처 + MVVM 패턴 프로젝트 스타터.

---

## 작업 시작 전 체크리스트

사용자의 작업 요청을 받으면 즉시 실행하지 말고, 먼저 다음을 확인한다.

1. **요청 목록 나열 및 확인**: 사용자가 요청한 항목을 번호 목록으로 정리하여 보여주고, 빠진 게 없는지 확인받은 후 작업을 시작한다.
   - 요청이 여러 메시지에 분산되어 있어도 반드시 하나의 목록으로 통합한다.
   - **명시적 진행 승인 문구**: "진행해줘" / "맞아" / "응" / "ok" / "go" 중 하나가 있어야 작업을 시작한다.
   - **판단이 애매한 경우**: "진행할까요?"로 한 번 더 확인한다.
2. **브랜치 확인**: 현재 브랜치가 main이나 develop이면 워크트리부터 생성한다.
3. **기존 구현 존재 확인**: 유사하거나 동일한 구현이 이미 존재하는지 확인한다.
   - `git log --oneline --all | grep {키워드}`
   - `ls src/presenter/component/{feature}/`
   - `grep -rn {식별자} src/`
   - 기존 구현이 있으면 사용자에게 알리고 덮어쓸지 확인한다.
4. **작업 유형 확인**: 새 기능 생성이면 설계자 소환, 기존 수정이면 작업 규모 판단으로 진행.
5. **파일 영향 범위 확인**: 어느 파일을 수정/생성하는지 미리 파악.

이 체크리스트는 작업 요청이 단순해 보여도 생략하지 않는다.

## 아키텍처

### 데이터 흐름
```
View → ViewModel(훅) → Domain → UseCase → Repository → API
```
이 방향만 허용. 역방향 절대 금지.

### 3파일 세트 (권장 패턴)
| 파일 | 역할 |
|---|---|
| `{Feature}Component.tsx` | 진입점. page.tsx에서 import |
| `{Feature}ViewModel.ts` | 비즈니스 로직 훅 |
| `{Feature}View.tsx` | 순수 렌더링 |

### 파일 크기 가이드
- 단순 View: 300줄 이내
- 복합 View: 600줄 이내
- ViewModel: 300줄 이내
- 공용 컴포넌트: 250줄 이내
- Entity: 150줄 이내
- 함수 80줄, JSX return 50줄 이내

---

## 핵심 금지 사항

- View에서 Repository/UseCase 직접 호출 금지 → 반드시 ViewModel → Domain → UseCase → Repository 순서
- Domain 인스턴스를 매 렌더마다 new 하지 말 것 → `useState(new SomeDomain())`
- `calc(100vh - ...)` 금지 → flex 구조 사용 권장
- page.tsx에 `'use client'` 사용 금지 → Component 파일에서만

## 기존 기능 보존 규칙

리팩토링·파일 정리 작업에서 Claude가 **단독 판단으로 기능을 사라지게 한 사례**가 누적되었다. 다음 규칙으로 재발을 차단한다.

### 삭제·단절 전 확인 의무
- 기존에 존재하던 컴포넌트·함수·prop·UI 블록을 제거하거나 import 체인에서 끊기 전에 반드시 사용자에게 확인받는다.
- "사용되지 않는 것 같다"는 Claude의 단독 판단 금지.
- 파일을 지우지 않고 "import만 빼서" 사실상 기능 제거한 경우도 동일한 확인 대상이다.

### `_` prefix 임시 조치 금지
- prop을 받지만 쓰지 않기 위해 `_` prefix로 표시하는 것을 허용하지 않는다.
- **"쓸 거면 연결하고, 안 쓸 거면 prop 자체를 삭제한다."** 중간 상태 금지.

### 리팩토링 시 회귀 체크리스트 의무
- PLAN.md에 **"영향받는 기존 기능 목록"** 섹션을 의무 작성한다.
- 커밋 메시지에 "제거" 또는 "단순화"가 포함되면, 본문에 무엇을 왜 제거했는지 명시한다.

### Lint warning 관리
- 신규 warning 발생 시 해당 작업 안에서 즉시 해소한다. 다음 작업으로 미루지 않는다.
- `@typescript-eslint/no-unused-vars` warning은 "잃어버린 기능"의 단서일 수 있으므로 발견 시 원인을 확인한다.

## Git Worktree 규칙

### 필수 조건
- 모든 feature 작업은 반드시 워크트리에서 진행한다.
- main, develop 브랜치의 작업 디렉토리에서 직접 코드를 수정하지 않는다.
- 이 규칙에 예외는 없다.

### 워크트리 라이프사이클

```bash
# 1. 생성
./scripts/worktree.sh create {name}
cd .worktrees/{name}
pnpm install

# 2. 작업 및 커밋
git add {파일}
git commit -m "feat: 작업 내용"

# 3. 머지 (원본 디렉토리에서)
cd ../..
git merge --ff-only feature/{name}

# 4. 제거 (머지 이후에만)
./scripts/worktree.sh remove {name}
```

### 자동 차단
develop/main 브랜치에서 파일 수정 도구(Edit, Write, Update)는 PreToolUse hook에 의해 물리 차단됨.

## Git Flow

### 브랜치 규칙
- `main`: 직접 커밋 절대 금지
- `develop`: 직접 커밋 금지. feature에서만 머지
- `feature/{작업명}`: develop에서 분기 → 작업 → 커밋
- `hotfix/{장애명}`: main에서 분기 → main + develop 둘 다 머지

### 커밋 메시지
- `feat:` / `fix:` / `refactor:` / `chore:`

### 브랜치 운영 원칙
- **머지는 사용자 명시적 승인 필수.** 기술적 검증 완료는 머지 승인이 아니다.
- **"작업 완료" 선언 권한은 사용자에게 있다.**
- **사용자 피드백은 기존 브랜치에 추가 커밋으로 반영.** 한 기능 = 한 브랜치.

## 에이전트 팀 운용

### 기본 구조
메인 세션이 팀장 역할을 겸한다. 서브에이전트는 메인의 전문 조수로, 요청받은 작업만 수행 후 세션을 종료한다. 서브에이전트끼리 직접 소통하지 않고, 메인을 통해 정보가 흐른다.

### 작업 유형 선행 판단

**새 기능/새 페이지 생성인가?**
→ 규모와 무관하게 설계자 소환하여 대화 기반 설계 진행. 이후 PLAN.md 작성 → 담당자 소환.

**기존 코드 수정이면:**

| 판단 기준 | 실행 방식 |
|----------|----------|
| 파일 1~2개, 한 영역 | 메인이 직접 처리 |
| 파일 3개 이상, data/ 영역만 | 개발자d 소환 → 리뷰어 |
| 파일 3개 이상, presenter/ 포함 | 담당 개발자 → 리뷰어 + 검토자 |
| 여러 영역 또는 구조 영향 | 설계자 → PLAN.md → 담당자 |

### 서브에이전트 소환 전 PLAN.md 체크리스트 의무화
서브에이전트를 소환하는 모든 작업에서, 소환 전에 PLAN.md에 태스크 체크리스트를 작성한다. 서브에이전트 완료 보고 수신 후 체크리스트와 1:1 대조하여 미완료 항목을 확인한다.

## 테스트 분류 기준

### 결정 트리
```
테스트 대상이 뭔가?
│
├─ DOM이 필요 없는 순수 함수/상수/타입인가?
│   → Unit 테스트 (Vitest)
│
├─ 단일 컴포넌트/훅의 인터랙션을 검증하는가?
│   → Component 테스트 (Vitest + Testing Library)
│
└─ 여러 페이지에 걸친 사용자 여정인가?
    → E2E 테스트 (Playwright)
```

**경계 판단 기준**: "전체 앱을 띄워야만 검증 가능한가?"
아니면 Component, 그래도 아니면 Unit. E2E는 마지막 수단이다.

### E2E 범위 제한
E2E는 **"이 화면에서 이거 누르면 이게 보인다"** — 이것만 검증.
컴포넌트 내부 동작(드래그 정밀도, 스크롤 발생 여부, 픽셀 단위 검증)은 Component 테스트로 내린다.

## 컨텍스트 관리 규칙

### PLAN.md 기반 상태 관리
에이전트 팀이 협업하는 작업에서는 `PLAN.md`를 작업의 상태 저장소로 사용한다.

**위치**: `.worktrees/{feature명}/PLAN.md`

**필수 섹션**: Context, 영향 범위, 단계별 분배, 제약 사항, 완료 조건, 현재 상태, 진행 이력

### /compact 제안 타이밍
- feature 브랜치를 develop에 머지한 직후
- 하나의 페이지/컴포넌트 구현이 끝나고 다음 작업으로 넘어갈 때
- 에이전트 팀 작업에서 Phase 하나가 완료되고 다음으로 넘어갈 때

### /compact 금지 타이밍
- 코드 구현 도중
- 여러 파일을 동시에 수정하는 리팩토링 중

## 모델 선택 기준

기본 모델은 Sonnet이다. 아래 기준에 따라 Opus 전환을 제안할 것.

### Sonnet으로 충분한 작업
- 3파일 세트 보일러플레이트 생성
- 단일 컴포넌트 버그 수정
- 스타일링 조정
- 커밋 메시지 작성, 단순 리팩토링

### Opus 전환을 제안해야 하는 작업
- 3개 이상 파일의 맥락을 동시에 이해해야 하는 작업
- 여러 계층을 넘나드는 디버깅
- 기존 코드 구조 분석 + 리팩토링 방향 판단
- 아키텍처 상담

**요약: 파일 1~2개면 Sonnet, 3개 이상 파일의 맥락을 동시에 이해해야 하면 Opus.**

## 디자인 리뷰

### UI 변경 시 필수 절차
1. 코드 작성 완료
2. verify 훅 자동 실행 (tsc + lint)
3. `.claude/design-review.md` 체크리스트 확인

### AI 슬롭 방지 규칙
- 이모지: UI 컴포넌트 내 사용 금지
- 그림자: `shadow-sm`까지만. `shadow-md` 이상은 사유 명시.
- 애니메이션: 접기/펼치기(150ms), 모달(200ms)만 허용. 장식용 금지.
- 라운드: 프로젝트 규정 값만 허용. `rounded-full` 등 과도한 라운드 금지.
- 과도한 그라데이션 금지.

## 작업 일지

### 파일 구조
- 위치: `docs/work-log/{YYYY-MM}.md` (월별 단일 파일)
- 파일 안에서 주차별(`W{번호}`)로 구분, 날짜별로 기록

### 기록 타이밍
develop 머지 수행 직후 기록.

### 기록 형식
```markdown
## W{주차} ({MM/DD} ~ {MM/DD})

- **{MM-DD} {feature 브랜치명}**: {완료 내용 한 줄 요약}
```
