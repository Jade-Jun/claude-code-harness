# 설계 결정 로그

> 이 파일은 프로젝트의 주요 설계 결정과 그 근거를 기록한다.
> 번호 기반으로 추적하며, 새 결정은 마지막 번호 + 1로 추가한다.

---

## 1. 브랜치 보호 방식 — 물리적 차단 채택

### 배경
CLAUDE.md에 "develop/main에 직접 커밋 금지"를 명시했으나, Auto 모드에서 반복적으로 무시되는 사례 확인.

### 결정
판단형 룰 대신 PreToolUse hook으로 Edit/Write/Update 도구를 물리적으로 차단.

### 근거
- 룰 기반 제어는 컨텍스트가 길어지면 우선순위가 밀림
- Auto 모드에서는 "이번만 예외"라는 임의 판단이 발생
- 물리적 차단은 예외 없이 작동

## 2. 에이전트 팀 역할 분리

### 배경
단일 에이전트에게 설계 + 구현 + 검증을 모두 맡기면 자기 코드를 자기가 검증하는 구조가 되어 품질이 떨어짐.

### 결정
6명 역할 분리: 설계자(Opus) + 개발자d + 개발자f + 리뷰어 + 검토자 + 테스터(조건부).
메인 세션이 오케스트레이터 겸 팀장.

### 근거
- 설계와 구현의 분리: 설계자가 방향을 잡고, 개발자가 구현
- 구현과 검증의 분리: 만든 사람이 아닌 다른 에이전트가 검증
- 영역 분리: data 계층과 presenter 계층의 소유권이 다르면 상호 침범 방지

---

## 3. PLAN.md 기반 상태 관리

### 배경
컨텍스트가 길어지면 초반 작업 상태를 잃고, /compact 후에는 진행 상황이 완전히 사라짐.

### 결정
작업 상태를 컨텍스트에 누적하지 않고 PLAN.md 파일에 기록. 에이전트 간 핸드오프도 PLAN.md를 통해 수행.

### 근거
- 파일은 컨텍스트와 독립적으로 영속
- /compact, /clear 후에도 PLAN.md를 읽으면 맥락 복구 가능
- 에이전트 간 직접 소통 없이 문서로 핸드오프하면 정보 손실 최소화
