# 에이전트 팀 운용 가이드

## 단계별 추천 조합

| 단계 | 소환할 서브에이전트 | 비고 |
|------|---------|------|
| 새 기능 (구조 영향) | 설계자 → 개발자d/f | 설계자와 메인이 대화 후 PLAN.md 작성 |
| UI 전용 구현 | 개발자f + 리뷰어 + 검토자 | data 변경 없는 경우 |
| 데이터 전용 변경 | 개발자d + 리뷰어 | UI 변경 없는 경우 |
| UI + 데이터 복합 | 개발자d + 개발자f + 리뷰어 + 검토자 | d 선행, f 후속 |
| 릴리즈 전 검증 | 리뷰어 + 검토자 + 테스터 (병렬) | 테스터는 이 단계에만 소환 |

## 설계 단계 흐름

1. 메인이 설계자에게 작업 브리핑
2. 설계자가 1차 제안 (확신 수준 명시)
3. 메인이 검증 및 대화 (여러 턴 가능)
4. 합의 후 메인이 PLAN.md 작성
5. 설계자 세션 종료

## 구현 단계 흐름

1. 메인이 PLAN.md 기반으로 담당자 소환
2. 담당자가 PLAN.md 읽고 작업
3. 구현 중 PLAN.md와 다르면 메인에 보고
4. 완료 후 PLAN.md 진행 이력에 기록

## 검증 단계 흐름

1. 메인이 검증자 소환 (View 변경 → 검토자 + 리뷰어, data만 → 리뷰어)
2. 각 검증자는 자기 영역만 검증 (중복 방지)
3. 통과 → PLAN.md 기록. 반환 → 사유 기록 후 메인이 담당자 재소환.

## 반환 규칙

- 검토자 반환 (🔴) → 개발자f 재소환 → 수정 후 검토자 재검증
- 리뷰어 반환 (❌) → 담당자 재소환 → 수정 후 리뷰어 재검증
- 같은 사안으로 2회 이상 반환 → 설계자 재소환하여 근본 원인 검토

## 핸드오프 원칙

서브에이전트 간 직접 소통은 하지 않는다. 모든 정보는 메인 세션과 PLAN.md를 거쳐 흐른다.
